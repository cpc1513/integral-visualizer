# 积分计算台正确性加固设计

## 目标

公开部署前优先消除“界面看起来成功、数学结果却对应旧输入或被改写区域”的静默错误。此次不增加收藏、图层等新功能，也不重做页面布局。

## 单一数据源

- 结构化 `IntegralSpec` 是计算与可视化的唯一权威数据。
- “完整公式”只有在能完整解析并同步到结构化字段时才算有效；解析失败时保留用户输入用于修正，但明确显示错误并禁止计算。
- 任何题目载入、积分类型切换或字段编辑都会使在途计算失效。结果只有在请求对应的规范指纹仍与当前规范一致时才能写回。
- 示例答案不再仅凭公式字符串恢复；用户状态不能借用默认题答案。

## 数学边界

- `oo`、`-oo`、`zoo`、`nan`、非实数结果属于发散或未定义，不能自动降级为数值积分。
- 条件区域数值积分只能把区域外样本置零；区域内的 `NaN/Inf` 必须报告定义域或奇点错误。
- 二重、三重积分中的孤立等式定义的是零测集，不能用随扫描范围变化的“厚带”冒充面积或体积。
- 上下界和参数形式只有在能证明等价时才提供一键条件转换；无法安全推导时保留原形式。

## 可靠性与恢复

- 本地状态带 schema 版本并经过运行时校验，损坏或旧数据回退到默认值；写入失败不影响计算台运行。
- Pyodide 初始化失败可重试，致命 Worker 错误会重建 Worker；计算有明确的超时与停止路径。
- 单点非有限采样只形成区域外点或曲线断点，不应让整幅合法图形失败。

## 性能与发布

- 数学输入使用延迟预览，避免每次按键立即触发高密度网格。
- Three.js 静止或离屏时停止持续渲染。
- 生产构建不公开携带完整 source map；后续再按实际 trace 定制 Plotly 包。

## 验收

- 单元/组件测试覆盖：公式解析失败、编辑期间旧结果返回、持久化损坏、发散积分、条件区域定义域错误、等式区域拒绝、安全转换与非有限采样。
- `pnpm test` 与 `pnpm build` 通过。
- 浏览器核心流程覆盖五类积分、题库载入、典型二维不等式区域、三维球体以及桌面/移动视口；若自动浏览器运行时不可用，明确记录未验证项。
